# 03 绑架问题：当 2D 导航失忆时，我让 3D 地图出来兜底

前面几篇里，我一直都在强调一个立场：  
主导航栈我还是更愿意交给 2D。`SLAM + Nav2` 足够成熟，代价也足够低。

但这不代表 3D 没用。  
3D 真正值钱的时候，往往不是平时导航，而是**2D 那套东西突然失忆的时候**。

这就是经典的 kidnapped robot problem。  
机器人被人搬了一下，或者里程计和局部定位突然彻底跑偏，`AMCL` 还以为自己在原地附近，`Nav2` 也还在按旧位置思考。这个时候继续相信 2D，系统只会越来越拧巴。

我这里的思路很直接：  
平时让 2D 栈负责导航，出事的时候，让 `RTAB-Map` 这条 3D 记忆/重定位线出来兜底。
![alt text](assets/03_kidnapped_robot_problem_demo_01.gif)
## 我实际做的，不是让 3D 取代 2D，而是让它专门负责“全局认祖归宗”

这条链的核心其实就三步：

1. 先用 `rtabmap_hybrid.launch.py` 建一份带 RGB-D 记忆的 3D 地图数据库  
2. 需要绑架恢复时，用 `rtabmap_localization.launch.py` 进入只读定位模式  
3. 再用 `rtabmap_pose_overwrite.py` 去比较 `RTAB-Map` 和 `AMCL` 当前到底差了多远

这里最关键的不是“3D 地图存在”，而是**3D 地图能不能在系统失忆时给出一个全局上说得通的位置**。

`rtabmap_pose_overwrite.py` 这部分我写得很克制。它不是看见 `RTAB-Map` 有结果就立刻强推，而是先做几层判断：

- 比较 `/rtabmap/localization_pose` 和 `/amcl_pose`
- 看平移差是不是超过阈值
- 看朝向差是不是超过阈值
- 如果 `AMCL` 协方差本来就已经很差，就把阈值再放低一点
- 再加一个最小重发间隔，避免 `/initialpose` 被疯狂刷屏

只有真的看起来像“2D 这边已经认错家了”，我才会把 `RTAB-Map` 的全局位姿重新发回 `/initialpose`，把 `Nav2/AMCL` 拉回正确坐标系。

而且在发之前我还会先发一个零速度。  
这个动作很小，但很重要。因为机器人在认知错位的时候最不该做的，就是一边错一边继续开。

## 这条恢复链的完整状态，其实很简单

这章不需要讲得太复杂，它的工作流状态就是下面这条线：

`正常 2D 导航 -> 发生绑架/严重漂移 -> RTAB-Map 从 3D 地图里做全局重定位 -> recovery bridge 比较 3D pose 和 AMCL pose -> 判定差异足够大 -> 停车 -> 重发 /initialpose -> 2D 导航恢复`

所以这章真正想表达的点也很明确：

- 平时我不用 3D 来主导导航
- 但我一直保留 3D 地图这条更贵、也更全局的认知链
- 一旦 2D 定位彻底失真，3D 就不再是“锦上添花”，而是兜底层

还有一个我自己挺喜欢的小细节，是我给 `RTAB-Map` 又加了一层 `adaptive_tuner`。  
机器人基本静止时，它会偏向更快的重定位参数；机器人在运动时，它又会把匹配条件收紧一点，尽量减少错误 teleport。这一点很工程，但也很真实，因为绑架恢复最怕的不是慢，而是**认错地方还特别自信**。

当然，他还是有可能会乱飞，这就涉及到很经典的对称房间问题。

## 所以 3D 地图在我这里的角色，不是更炫的导航，而是最后的世界记忆

如果 Route A 讲的是对象地图，Route B 讲的是场景记忆，那这一小章讲的其实是另一种很实用的 3D 价值：

**当 2D 世界模型崩掉时，谁来告诉机器人它到底身在何处呢**

这也是为什么我一直不觉得 3D 和 2D 是非此即彼。  
在我这套系统里，2D 负责平时高效地活着，3D 负责在系统突然失忆时，把它从错误世界观里拽回来。
