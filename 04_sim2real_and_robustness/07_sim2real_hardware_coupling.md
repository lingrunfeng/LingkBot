# 07 跨越虚实：LeoRover 真机耦合，不是把仿真搬过去

## 到了真机这里，问题突然不再浪漫了

前面几篇写到这里，系统已经越来越像一套完整的机器人工作流了。  
仿真里，`SLAM`、`Nav2`、探索、语义导航、视觉接近、抓取，基本都已经被压进了同一条链里。

但一碰真机，事情的气质会立刻变掉。

在仿真里，很多问题是算法问题、参数问题、状态机问题。  
到了真机，最先冒出来的往往不是这些，而是更底层、更难看、也更不体面的东西：

- 串口设备到底是不是那台设备
- 雷达坐标系到底有没有接反
- `cmd_vel` 到底该不该直接放给底盘
- 激光数据里哪些是真的障碍，哪些只是现实世界的脏噪声
- 建图后端、控制器和底盘实际运动特性，到底有没有站到同一边

![alt text](assets/07_sim2real_hardware_coupling_figure_02.png)

所以这一篇我不想把它写成一篇“真机 demo 成功复刻”的文章。  
因为实话实说，到目前为止，真机这一侧我真正完整测试到的，主要还是**探索建图**这一段。更后面的巡逻、视觉接近、抓取，在软件架构上已经被接好了，但实验时间并没有允许我把整条链在真实 LeoRover 上从头到尾慢慢磨完。

说到这...实验室一周四个小时，真的不够。。。  
对真机系统来说，四个小时甚至连“稳定复现实验条件”都很紧张，更不要说还要调设备、重启节点、查 TF、盯 topic、改 launch、再重新跑一遍。很多时候不是系统不行，而是时间窗口本身就不允许它被慢慢磨平。

但反过来说，我对这条真机路线其实并不悲观。  
因为最难啃、也最没有观赏性的那块东西，我已经花了两周宝贵的实验室时间把它解决掉了：**硬件通信与接口耦合。**

也就是说，剩下的问题更多像调参与现场磨合，而不是整套系统到底能不能落地的结构性未知数。

![alt text](assets/07_sim2real_hardware_coupling_figure_01.png)

当然，如果时间不够那什么都白搭...

## 我后来越来越觉得，Sim2Real 本质上是接口工程

很多人讲 `sim2real`，喜欢把重点放在 domain gap、传感器噪声建模、视觉泛化这些听起来比较“研究”的词上。  
这些当然都重要。

但在我这个项目里，真正先把我拖进泥里的，其实不是这些，而是更直接的接口问题：

- 真实雷达驱动出来的 `frame_id` 和 URDF 里的链条对不上
- 真机雷达安装方向和我在模型里假设的方向并不一致
- 底盘并不适合吃一条完全裸奔的 `cmd_vel`
- 真实激光会有孤立噪点、短时动态遮挡、近距脏数据
- 真实探索时，建图跳一下，控制层就可能立刻开始抽

所以我后来给真机这一层做的事情，其实很不“炫”，但很关键：  
**不是重写上层任务逻辑，而是把所有会把上层逻辑搞死的接口缝，先一条条补上。**
**并且写一套完全隔离的专供真机使用的节点，完全不让真实和仿真互相污染**

这也是为什么我最后的真机 launch 架构，和纯仿真版本看起来长得像，但中间多了不少现实世界专属节点：

- `scan_dynamic_filter`
- `cmd_vel_guard`
- `map_jump_watchdog`
- `map_snapshot_manager`
- 真机专用 topic remap
- 真机专用 `URDF/xacro`
- 真机专用保护型 launch 组合

这些东西单独拿出来都不性感，但没有它们，后面那条漂亮的工作流很可能根本没有资格碰真机。

## 真机这里最先解决的，不是导航，而是设备身份

这件事很土，但真的是现实。

机器人真机调试里，一个特别早、也特别烦的问题，是设备到底能不能以**稳定身份**出现。  
如果雷达今天叫这个 `ttyUSB`，明天又跳到另一个口，整套 launch、驱动、脚本、启动顺序都会开始变得神经。

所以我最后专门补了一层 `udev` 规则，把 `RPLidar` 固定映射到 `/dev/rplidar`。  
这一步在文章里看起来像一句话，但实际上它背后代表的是一个很重要的判断：

**真机通信首先要解决“设备是谁”，然后才谈“数据是什么”。**

这也是我为什么说，最难的硬骨头之一其实已经啃下来了。  
一旦真实设备的身份稳定了，后面很多 launch、脚本、驱动入口才会真正开始变得可重复。

## 真机 launch 没有另起炉灶，而是把风险点显式插进主链

我没有把真机那套东西写成完全独立的另一套系统。  
相反，我尽量保留了和仿真一致的主链：

- `robot_state_publisher`
- `TF`
- `SLAM`
- `Nav2`
- exploration / patrol orchestration

真正变化的，是我在这条主链前后，插入了现实世界特有的几层保护和桥接。

比如真机主 launch 里，我显式保留了：

- `slam_toolbox` 和 `cartographer` 两条建图后端入口
- 真实激光的 `scan -> scan_filtered` 过滤链
- `cmd_vel_raw -> cmd_vel` 的受控输出链
- map jump 监控与紧急停车
- 地图快照与稳定地图备份

这背后的思路其实很简单：

**仿真和真机的差别，不应该体现在上层任务语义完全变掉，而应该体现在底层约束被更诚实地暴露出来。**

也正因为这样，我保留了很多“像开关一样”的 launch 参数。  
这不是为了炫配置多，而是因为真机上很多能力最好是可拆、可关、可替换的：

- 要不要先只建图，不开 `Nav2`
- 要不要先开 scan filter，再决定控速保护层是否联动
- 要不要切 `slam_toolbox`
- 要不要切 `cartographer`
- 要不要开 jump watchdog

真机不是适合一键梭哈的地方。  
它更像分层通电。每多开一层，就多接近真实能力，但也多一层潜在故障面。

![alt text](assets/07_sim2real_hardware_coupling_figure_03.png)

## 一个很小但很真机的问题：雷达方向反了，TF 就得老老实实认

这是专门对于我这个真实世界的细节，因为它特别能说明真机和仿真的差别。

在真机 launch 里，我专门加了一条 `scan_link -> laser` 的静态变换，而且 yaw 默认就是 `pi`。  
原因并不玄学，就是那台真实 `LiDAR` 的安装方向和模型假设并不一致，等于在车上是“反着装”的。

这件事在论文里根本不值一提。  
但在真实系统里，它会直接决定：

- 建图是不是朝对的方向长
- 障碍物是不是出现在对的方位
- `Nav2` 看见的世界是不是和底盘将要去的世界是同一个世界

我后来越来越接受一件事：  
**真机不会自动原谅错误的坐标系。**  
仿真里很多地方靠默认值、靠插件暗中兜底就能活，真机里坐标链一旦没站稳，上层就会变成漂浮建筑。

## 真实雷达不等于仿真雷达，所以我单独插了一层 scan filter

这一层我觉得挺有代表性。

在 Gazebo 里，激光经常是干净的、边界明确的、障碍物也比较“守规矩”。  
真机不是这样。

真实扫描里会混进来很多很烦的小东西：

- 过近的脏回波
- 过远的不稳定点
- 一闪而过的动态遮挡
- 孤立 speckle
- 几帧里才短暂出现一下的假障碍

所以我最后没有直接把原始 `/scan` 扔给 `SLAM` 和 `Nav2`，而是在中间插了一个 `scan_dynamic_filter`。  
这层做的事其实很工程：

- 先做近距和远距裁剪
- 再做时间窗口中值滤波
- 再去掉孤立 speckle
- 最后根据背景模型做一层动态目标抑制

说白了，就是尽量别让上层把那些“一闪而过、根本不应该被当成稳定世界结构”的东西，误认成地图的一部分。

这件事在真机里非常值。  
因为 exploration 和 Nav2 最怕的不是“完全看不见”，而是“看见一堆不该信的东西，还真的信了”。

## `cmd_vel` 这件事也一样，真机不能裸奔

仿真时我当然也知道速度不能乱来。  
但真机这里，我是明确把控制输出做成两段的：

- 上层节点写 `cmd_vel_raw`
- 真正给底盘的是经过保护层后的 `cmd_vel`

这中间那层 `cmd_vel_guard`，本质上就是一个很朴素的现实主义节点。它不讲梦想，只讲上限：

- 线速度上限
- 角速度上限
- 线加速度上限
- 角加速度上限
- 指令超时自动清零
- map jump 时直接刹停
- 某些模式下直接禁止倒车

我还写了 `forward_only` 这个真机探索配置。  
因为它背后的判断其实很清楚：在某些探索场景里，允许底盘自由前后折腾，并不一定更聪明，反而更容易把系统带进滑移转向平台最不稳定的动作区间。这里就是仿真和现实世界的区别，仿真可以随便折腾，真实世界可不行。

所以有时候“更保守”不是能力退化，而是我明确承认：  
**真机安全和状态可控，比多一个理论上的动作自由度更重要。**


## 真机探索用的不是“图最漂亮”的 SLAM，而是“闭环最不容易互相伤害”的 SLAM

这一段其实和前面仿真那篇呼应得很强。

在真机 mapping/exploration 这条线上，我最后还是更倾向 `slam_toolbox`。  
不是因为 `cartographer` 没价值，更不是因为它建图不行。恰恰相反，在某些条件下它的图是很好看的。

但真机 exploration 里，我越来越看重的是另一件事：

**图能不能持续长，控制链会不会被位姿修正反复拽坏。**

所以我后来专门给真实探索留了一份更保守的 `slam_toolbox` profile。  
里面一个特别关键的判断就是，在 exploration 阶段先把 `do_loop_closing` 关掉。

这听起来像退步。  
但其实不是。

因为这份 profile 更看重的是局部一致性，而不是在机器人一边动、一边试探边界的时候，让全局图优化突然来一次很激进的结构回拉。  
对探索来说，很多时候“稳稳地继续往前长”比“现在这张图看起来更完美”更重要。

这其实也是我在真机上学到的另一个教训：  
**真实机器人里，漂亮地图和稳定动作，并不总是同一件事。**

**安全第一**

## Nav2 到了真机，也不是照搬仿真参数

真机这边的 `Nav2` 参数我压得相对保守。

我最后更偏向这些东西：

- `RegulatedPurePursuit`
- 不允许总是倒车
- 先转向再前进
- 更大的膨胀层
- 更保守的机器人半径
- 更低的旋转速度
- 更明显的 recovery backup

这些设置说到底都在回答同一个问题：  
**如何把那种天然不那么“优雅”的运动学，尽量包进一个不会太神经质的导航行为里。**

LeoRover 这种底盘，并不是那种在原地小碎步调姿态会特别好看的平台。  
有时候控制器越激进，看起来越像在和底盘物理现实打架。

所以真机 Nav2 这层，我后来越来越像是在做一件有点灰色、但很真实的事情：  
把参数调成一种“带着敬畏感开车”的状态。

## 真机的工作流其实已经写好了，只是实验验证暂时停在探索建图边界

这一点我想写清楚，不然很容易被误解成“真机部分只做了零散测试”。

其实不是。

从代码结构上看，真机工作流已经不是碎片化脚本了，而是一条明确编排过的状态链：

1. 启动真机保护型建图/导航环境
2. 进入 `PinkBallExplorer` 探索阶段
3. exploration 完成后自动保存地图
4. 进入一个桥接式倒计时确认阶段
5. 再切入 patrol / visual interrupt 这一套后续逻辑

也就是说，**探索 -> 存图 -> 桥接 -> 巡逻** 这条线，在软件上已经被完整写出来了。

而且这里面有几个很有“真机味”的细节：

- exploration 支持外部 stop service / stop topic
- 检测到 map jump 会暂停探索并取消当前 goal
- 真机探索默认走 `/cmd_vel_raw`
- patrol 读的是 `/merged_odom` 和 `/scan_filtered`
- bridge 阶段不是硬切，而是给一段人工确认与缓冲时间

这些都说明，这一层已经不是“我以后大概会这么做”的草稿，而是一个已经按真实部署习惯编排过的状态机。

只是现实是，实验室时间真的不够让我把 patrol、视觉接近、抓取在真实平台上继续按同样粒度打磨到底。

这也是为什么这一章我愿意写得稍微晦涩一点。  
因为它真正吸引人的部分，确实不是一个特别好看的展示视频，而是这套从接口、约束、保护层、状态桥接到后续扩展位的**部署语法**已经开始成形了。

## 所以这一章真正想说的，不是“我上真机了”，而是“我把最难看的那一层啃开了”

我现在回头看，真机这部分最值钱的，不是某一次成功移动，不是某一张图建得多漂亮，也不是把某个 launch 跑起来这件事本身。

最值钱的是，我已经把很多真机会把人拖死的底层问题，开始系统化地收进同一套架构里了：

- 设备身份固定
- 真机 topic / frame 耦合
- 激光现实噪声过滤
- `cmd_vel` 保护与限幅
- map jump 监控与急停
- 探索阶段专用 SLAM 配置
- 真机保护型 exploration launch
- 探索到后续任务的桥接状态
还有
**ROS通讯**

这些东西不会让文章标题变得更炫。  
但它们会决定一套系统到底是“仿真作品”，还是“开始有资格碰真实机器人”。

如果前面几篇写的是能力进化，那这一篇更像一场降维。  
不是往上升，而是往下扎。

因为所有机器人项目，最后都得回答一个问题：

**系统离开理想世界以后，还能不能活。**
