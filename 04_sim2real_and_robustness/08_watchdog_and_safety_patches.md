# 08 真机不死之身：Watchdog、跳变保护与稳定地图补丁

## 如果说上一章写的是耦合，这一章写的就是别死

到了真机这里，系统会开始暴露出一种很尴尬的现实。

很多时候它不是彻底坏掉。  
它能动，能看，能建图，`Nav2` 也没死，探索逻辑也还在跑。  
但它会在某些瞬间突然出现一种很熟悉的症状：

- 地图像被人拽了一下
- 机器人动作突然变得犹豫甚至抽搐
- 正在往 frontier 走，结果又像被拉回去
- 前一秒还在稳定推进，后一秒局部控制开始别扭

这类问题最烦的地方在于，它不是那种一眼就能定位的红色报错。  
它更像一套本来已经能工作、甚至本来就挺鲁棒的系统，在真实世界里被一些离散的、短时的、但足够恶心的跳变反复打断。

所以这一章我想写的，不是什么“神奇新算法”。  
这一章本质上是一组补丁，一组 watchdog，一组很不浪漫但很值钱的保护逻辑。

它们干的事情其实非常朴素：

- 先承认地图会跳
- 再承认一旦跳了，机器人继续冲往往只会更糟
- 然后在那一瞬间把系统拉住
- 给建图和位姿估计一点重新稳定的时间
- 同时尽量保住最近一份还能用的地图

某种意义上，这一章确实在解决真机探索和建图时的跳变问题。  
但如果要说得更老实一点，它的核心思路可能并不玄妙，甚至有点像一句很土的话：

**很多鲁棒性，最后其实就是让机器人别那么急。**

这听起来像退步。  
但真实机器人里，知道什么时候该慢、该停、该等一口气，往往比多跑那几步更重要。

## 我后来越来越觉得，鲁棒性不是“永远不出错”，而是“错了先别放大”

这一点其实是我看自己代码时感受最强的地方。

很多所谓鲁棒系统，外面看起来很强，像是什么都能扛。  
但真正落到工程上，鲁棒性的第一步往往不是“系统不会错”，而是：

**系统一旦开始错，不要让这个错误立刻传播成更大的动作灾难。**

这也是我最后给真机这一层加 watchdog 的原因。  
我没有把问题理解成“我要消灭一切 map jump”。  
这事很不现实。

我更在意的是另一件事：

如果 `map -> odom` 在某个瞬间突然发生了超过阈值的位姿跳变，而机器人此时还在按旧认知继续往前冲，那后果通常比 jump 本身更糟。

因为这时系统内部会开始出现一种非常糟糕的错位：

- 上层 planner 以为机器人在 A
- 底盘真实物理位置可能已经更接近 B
- exploration 还在往刚才那个 frontier 下 goal
- controller 还在试图沿着上一刻的路径修正

只要这个错位继续被动作层放大，机器人就很容易进入一种“明明每个模块都还活着，但整体越来越不对劲”的状态。

所以我的做法不是很花哨。  
我只是把这个瞬间单独提出来，承认它是一类系统事件，然后给它一个完整的响应链。

## 这个补丁链的起点，不是恢复，而是检测 `map -> odom` 是否突然变了

整套保护最前面的那个节点，其实就是 `map_jump_watchdog`。

它干的事情不复杂，但思路很干净：

1. 周期性读取 `map -> odom` 变换
2. 计算相邻两次之间的平移变化和旋转变化
3. 如果超过阈值，就记一次 violation
4. 连续超过阈值若干次，才正式判定为 jump

这几个细节很重要。

第一，它盯的不是底盘速度，不是里程计本身，而是 `map -> odom`。  
也就是说，它关注的是**全局定位层在短时间内有没有出现不自然的离散修正**。

第二，它不是一次超阈值就立刻尖叫。  
我这里保留了 `consecutive_violations` 这种机制，就是不想让系统被偶发抖动轻易触发误报。

第三，它把平移阈值和旋转阈值拆开了。  
因为真实世界里，跳变并不总是只表现成位置猛挪一下，有时候是姿态突然被扭了一截，这对探索和控制一样致命。

我很喜欢这种写法。  
它不是那种“出了问题以后靠经验去猜”的补丁，而是明确把 jump 定义成：

**一个可观测、可计数、可阈值化的系统事件。**

## 一旦 jump 成立，我没有让系统硬扛，而是先停

这部分其实就是整个补丁链里最不花哨、但可能最重要的决定。

`map_jump_watchdog` 一旦判定 jump 成立，做的第一件事不是纠正，不是重规划，也不是继续赌。

它先做两件事：

- 向 `emergency_cmd_vel` 发布零速度
- 发布 `map_jump_detected = true`

也就是说，我先把“机器人别继续乱动”这件事，放在了“我要不要马上变聪明”前面。

这在机器人里其实很重要。  
因为 jump 发生的瞬间，最危险的不是系统短暂不准，而是系统在**不准的时候还充满自信地继续执行动作**。

所以这个 watchdog 里还有一个 `hold_stop_sec`。  
它的意思很朴素：不是检测到 jump 之后只刹一拍，而是让系统明确进入一小段“停住别动”的窗口。

这一小段时间，从宏观看很不起眼。  
但在建图、局部控制、TF、探索目标还在交织的系统里，它相当于给整条链一个机会，把那口被拉乱的气重新顺一下。

## 这套补丁最值钱的地方，不是刹车，而是它会沿着工作流往后传

如果只是单独停一下车，那还只是个安全开关。  
它真正开始变得有系统味，是因为这个 jump 事件没有停在 watchdog 自己那里。

它继续往后传播了。

先是 `cmd_vel_guard` 会订阅 `map_jump_detected`。  
也就是说，就算上层还有某些指令残留，保护层也会在 jump 期间主动把目标速度压成零。

然后是真机探索节点。  
不管是 `pink_ball_explorer.py` 还是 `explore_and_patrol_real.py` 里的真实探索包装，都专门订阅了这个事件。

它们在 jump 发生以后做的不是“忽略一下继续跑”，而是：

- 暂停探索若干秒
- 取消当前 Nav2 goal
- 等暂停窗口过去以后，再恢复目标选择和继续探索

这就很关键了。

因为 exploration 的本质不是纯反应控制，它是一条**不断生成目标、不断推进未知边界**的状态链。  
如果地图刚刚跳过一次，结果探索器还抱着刚才那个 frontier 不放，那它后面生成出来的很多动作，逻辑基础已经不干净了。

所以我最后不是只给底盘加 brake。  
我其实是把 jump 事件接到了整条探索工作流里，让它变成：

`检测异常 -> 刹车 -> 暂停 -> 取消旧目标 -> 等图稳 -> 再继续`

这一点我认为是这章真正的精华。

## 其实这已经不只是“防跳变”，而是在给探索状态机加一个短暂的失重保护层

这一点我很想写出来。

因为如果只从结果看，很容易把这章理解成：

- 检测跳变
- 机器人停一下
- 所以系统稳了一点

这当然没错，但还不够。

更准确一点说，这组补丁其实是在 exploration 这条主状态机外面，再包了一层很薄的保护逻辑。  
它不改变探索本身的目标，不改变 frontier 提取方法，也不改变 Nav2 的基本职责。

它只是承认：

**当全局定位层出现离散扰动时，探索状态机应该暂时失去行动权。**

这句话听起来有点绕，但确实很有用呀。

因为很多系统会把 exploration 当成一种必须持续推进的东西。  
而我后来越来越觉得，持续推进本身不是目的，**在脏状态下不继续放大错误**才是前提。

所以这章看起来像安全补丁，实际上它已经在轻微地改变整套系统的节奏了：

- 系统不是一直向前冲才算强
- 系统知道什么时候应该沉默，反而更强

## `map_snapshot_manager` 这条线也有价值，它不是救当前，而是在保最近一份还能信的地图

除了 watchdog 以外，我觉得还写了一条很漂亮的线，就是 `map_snapshot_manager`。
这里真的很酷，我真觉得我很有想象力

它做的事也不复杂：

- 定时触发地图快照保存
- 保留最近若干份历史地图
- 始终更新一份 `latest_stable_map`
- jump 刚发生或刚结束的一段时间里，暂停频繁保存

这套逻辑的价值，不在于它能当场修复什么。  
它更像是在说：

**如果当前地图正在经历不稳定阶段，那我至少要保住最近一份还算稳定的结果。**

这在真实环境里非常合理。

因为真机探索不是一次函数调用，而是一个持续过程。  
地图不是“生成一次就永远正确”，而是在一段时间里不断长、不断修、偶尔还会被局部拉扯。

所以这个 snapshot manager 背后的想法其实挺前卫的：

- 平时周期性存档
- 跳变期间不瞎存
- 跳变后等一个 cooldown，再恢复
- 始终给系统留一份最近的稳定地图出口

它不是那种炫目的功能。  
但它会让整个系统从“当前这一刻是不是还活着”，变成“就算这一刻有点乱，最近稳定成果也没丢”。

## 从完整工作流看，这一章补的不是一个点，而是一整段“出问题以后怎么办”

如果把前面几篇文章和这一章连起来看，整个真实工作流其实会变得更完整。

原来那条线更像是：

`bringup -> SLAM/Nav2 -> exploration -> save map -> bridge -> patrol`

而这章补完以后，它更像是：

**bringup -> SLAM/Nav2 -> exploration -> jump detect? -> emergency stop -> pause/cancel -> wait stable -> resume exploration -> save stable map -> bridge -> patrol**

也就是说，系统里终于开始出现“异常分支”了。

这点很重要。  
因为没有异常分支的工作流，看起来很顺，但本质上只是顺风局。  
一旦地图跳一下、位姿扭一下、控制突然不对劲，整个系统就会暴露出它到底有没有第二层思考。

而这章做的，就是把“第二层思考”写进工作流里。

它没有让机器人突然变成什么高智能体。  
它只是让系统第一次认真地回答：

**如果当前认知层自己不稳定，动作层应该怎么办。**

## 这组补丁是不是本质上就是“让机器人跑慢点”？

确实，某种程度上，是的。

至少从效果上看，这组补丁做的很多事情，本质上都在朝一个方向收：

- 更谨慎地承认 jump
- 更快地刹住车
- 更慢地恢复动作
- 更保守地继续探索

如果要刻薄一点说，它确实像是在把系统调得更怂。  
但我并不觉得这有什么问题。

因为真机鲁棒性很多时候本来就不是靠“更猛”得到的，而是靠更少的误动作、更少的误自信、更少的错误放大。

而且从另一个角度看，我本来的系统其实并不脆。  
前面那整套 `SLAM + Nav2 + pink exploration` 本来就已经有一定鲁棒性了，不是那种轻轻一碰就彻底散架的东西。

所以这一章的意义，不是从零拯救一个完全不行的系统。  
更像是在一套本来就能工作的系统上，再补一层**让它在真实环境里更不容易被离散扰动扯坏**的保护皮肤补丁。

这就够啦。

## 我后来越来越接受，真实机器人里的很多高明补丁，外观看起来都像“克制”

这可能是这一章最后真正想说的话。

很多人以为的精彩，一定长得像更复杂的算法、更聪明的推理、更高级的模型。  
但我现在回头看，我设计这一组 watchdog 和 safety patch ，恰恰不是它们多复杂。

而是它们很克制。

它们做的事情非常有限：

- 看到不对劲，就先停
- 停了以后别装作没发生过
- 等一小会儿，让图稳一稳
- 把稳定地图留住
- 然后再把探索权交回去

这整套逻辑说穿了，没有神话。  
但它会直接决定一套真机系统到底是“偶尔能演一次”，还是“开始有一点长期活下去的样子”。

如果上一章回答的是“怎么把系统接到真实 LeoRover 上”，  
那这一章回答的就是另一件更沉一点的事：

**接上去以后，系统怎么别因为一点跳变就把自己扯坏。**

当然，不知道是不是心理原因，一周前还跳变的真机导航测试，一周之后上了补丁他就不跳了，不知道是我调慢了车的速度还是这个补丁奏效了，我无法确定是否是这个补丁奏效了，但从结果上看，如果他本来就可以在不打补丁的情况下生存，补丁不生效也挺好。就像一个汽车安全气囊，你总不会希望他爆了吧，但它得在。
